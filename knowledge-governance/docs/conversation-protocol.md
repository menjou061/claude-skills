# Conversation Protocol — 對話接口協議

> 對話是 L0 統一接口，所有採集源都從對話進入。
> 對話本身也是知識生成過程，需要沉澱否則歸零。

---

## 1. 4 種對話流模式

### A. 貼料流（採集流）

**識別信號：**
- 「這篇文章 / 這本書 / 這段播客...」
- 用戶貼 URL / 文件路徑 / 大段引用文字
- 用戶說「我看了 X，想討論」

**處理路徑：**
1. AI 識別來源類型（書/復盤/片段/工作產物）
2. 引用對應採集源 SOP（ADR-005~008）
3. 詢問或自動應用覆蓋標記
4. 進入該源的處理流程

**示例：**
```
用戶：[tier=A] 我讀完了 CZ 幣安人生，幫我提取規則候選
AI：識別為書籍採集（Tier A）→ 進入 ADR-005 流程 → 提取候選 → 走 7 閘
```

---

### B. 設計流（架構流）

**識別信號：**
- 「我們設計一個 X」「建立 Y 機制」
- 「我想加一條 ADR」「需要一個 SOP」
- 用戶描述問題場景，希望系統化解決

**處理路徑：**
1. AI 識別為設計流，提示「進入 B 模式」
2. 引導用戶說明：Context（問題） / Decision（方案） / Consequences（後果）
3. 起草 ADR 草稿，用戶確認後寫入 `docs/decision-records/`
4. 更新 SKILL.md 索引和 CHANGELOG

**配額**：月度 5 個 ADR 提案上限。

---

### C. 諮詢流（應用流）

**識別信號：**
- 「我該怎麼做 X」「Y 違反規則嗎」
- 「我打算 Z，幫我看看」
- 用戶在決策前尋求規則庫意見

**處理路徑：**
1. AI 識別為諮詢流，retrieve 對應域 rules.json + 跨域原則
2. 對用戶意圖逐條對照規則
3. 標記 ✅ 符合 / ❌ 違反 / ⏭️ 不適用 / ❓ 信息不足
4. 給出建議，不寫入任何文件（默認 L0 歸零）
5. 若用戶說「以後都這樣」→ 觸發 L2 結晶為規則候選

**示例：**
```
用戶：我想加倉 [Stock A] 5 股，現在價格 $XXX，幫我看看
AI：諮詢流 → retrieve trading-discipline → 對照 P1-09/P1-10/P1-11 → audit 報告
```

---

### D. 反思流（復盤流）

**識別信號：**
- 「我剛踩了個坑」「總結一下我這次...」
- 「最近一週/月的操作」
- 用戶帶著情緒或時間範圍說「該復盤一下」

**處理路徑：**
1. AI 識別為反思流，進入 ADR-002 復盤 SOP
2. 引導用戶說：事件 / 違反規則 / 印證規則 / 新候選
3. 寫入對應域的復盤文件（如 `01-財經/復盤/`）
4. 自動提取規則候選（1-2 條/篇）

---

## 2. 模式混合處理

對話經常混合多種流。AI 必須在模式切換時主動提示：

```
用戶：[諮詢] 我想加倉 [Stock A]... [反思] 對了我上週賣 [Stock B] 卖飛了
AI：檢測到模式切換 諮詢流 → 反思流
    建議：先完成 [Stock B] 反思流（觸發 P1-09 命中？），再進諮詢流評估 [Stock A]
    您希望按哪個順序處理？
```

## 3. 4 層沉澱機制

### L0 即時（自動，歸零）

- 對話本身在 chat 上下文中
- chat 結束後消失
- **不寫入 memory**，避免噪音

### L1 會話摘要（每場 chat 結束自動）

**觸發**：每場 chat 結束時 AI 自動生成
**位置**：`docs/chat-logs/[YYYY-MM-DD]-[主題].md`
**格式**：

```markdown
## [YYYY-MM-DD] [主題]
- 流模式：A/B/C/D（混合則列多個）
- 核心 takeaways：3-5 條
- 結晶產出：[ADR/規則/decision] 列表
- 待結晶：[列表，給用戶下次回顧]
- 關聯 Thread：[active-threads.md 中的 Thread 名]
```

### L2 結構化結晶（識別信號時主動寫）

**觸發信號 → 結晶類型**：

| 信號 | 結晶類型 | 寫入位置 |
|------|---------|---------|
| 「以後都這樣做」 | 規則候選 | `domains/*/rules-candidates.json` |
| 「我決定 X」 | 決策記錄 | `docs/decisions/[日期]-[主題].md` |
| 「設計一個 X 機制」 | ADR 候選 | `docs/decision-records/[編號]-[主題].md` |
| 「下次遇到 Y 我會 Z」 | SOP 補充 | 對應 SOP 文件 |
| 「我以前一直 X 是錯的」 | 規則候選（負向） | `domains/*/rules-candidates.json` |

**用戶顯式觸發**：用戶可用 `[save-as=adr|rule|decision]` 強制結晶。

### L3 長期記憶（Active Threads）

**位置**：`memory/active-threads.md`
**內容**：跨 session 連續性的主題追蹤
**詳見**：[[ADR-009]] + Step 3 機制

## 4. 配額管控

| 配額 | 上限 | 超限處理 |
|------|------|---------|
| 單場對話晉升候選 | 3 條 | 超出進 parking-lot |
| 月度 ADR 提案 | 5 個 | 超出延後 |
| 月度規則候選（對話來源） | 10 條 | 超出延後 |

---

## 5. AI 工作流

每場 chat 開始時：

1. **加載 Active Threads**：讀 `memory/active-threads.md`
2. **第一條回應簡報 Thread 狀態**：「上次中斷點是 X，下一步是 Y」
3. **識別流模式**：根據用戶首條消息判斷 A/B/C/D
4. **應用對應 SOP**：見上方各流的處理路徑

每場 chat 結束時：

1. **自動生成 L1 摘要**：寫入 `docs/chat-logs/`
2. **詢問是否更新 Thread**：「需要更新 Thread [名稱] 的狀態嗎？」
3. **檢查待結晶**：若有 L2 結晶信號但未處理，提示用戶

---

## 6. 紅線

1. ❌ L0 對話本身寫入 memory（污染）
2. ❌ 跳過模式識別直接執行
3. ❌ 結晶無類型標記
4. ❌ 單場超配額仍強行寫入
5. ❌ ADR 提案不走 ADR-001~004 標準結構
