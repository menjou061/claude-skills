# ADR-008 工作產物雙向流動 + 預設機制（Work-Artifact Bidirectional Flow + Defaults）

## Status
Accepted (2026-05-31)

## Context

PRD、需求背景、技術方案這類工作產物是「正在進行的決策」，與其他採集源（書/復盤/片段）有根本差異：
- 書是過去的、外部的、已完成的提案來源
- 復盤是過去的、自我的、已完成的提案來源
- **工作產物是當下的、自我的、進行中的決策對象**

兩個矛盾：

**矛盾 1：方向錯置**
- 自然反應是「從 PRD 中提取規則候選」（與書一樣的處理）
- 但實際應該反過來：**規則庫 audit PRD**（檢查 PRD 是否違反既有規則）
- 工作產物的主流向是 L5 應用層（規則→PRD），不是 L1 採集層（PRD→規則）

**矛盾 2：每次填表負擔**
- 6 類採集源各有不同的默認動作 / 配額 / 閘配置
- 用戶每次採集都要填一遍 = 摩擦極大
- 需要「最低破壞性默認 + 顯式覆蓋」機制

## Decision A：工作產物 SOP

### 主流：PRD audit（L5 反向）

| 步驟 | 操作 |
|------|------|
| 1 | 從 rules.json 拉 product-experience 全部 + cross-domain 5 條 |
| 2 | 對 PRD 逐節掃描，每條規則對應一個 audit 標記 |
| 3 | 標記 ✅ 符合 / ❌ 違反 / ⏭️ 不適用 / ❓ 信息不足 |
| 4 | 輸出 audit 報告到 `docs/audit/[PRD名]-audit-[日期].md` |
| 5 | 不修改 PRD 原文 |

### 次流：背景想法提案（快速通道）

- PRD 中的「我意識到 X」「下次應該 Y」型語句 → 提取為候選
- 配額：**0-2 條/份**（嚴格限制）
- 流程：走手寫快速通道（4 閘：0+1+2+5）

### Strict 模式（audit 嚴格度）

| 模式 | P1 | P2 | P3 | 跨域 |
|------|----|----|----|------|
| strict=high | 必修 | 必修 | 必修 | 必修 |
| **strict=mid（默認）** | 必修 | 建議 | 忽略 | 必修 |
| strict=low | 必修 | 忽略 | 忽略 | 建議 |

### 紅線

- **不修改 PRD 原文**，只產 audit 報告
- **不寫 rules.json**，候選只進 candidates.json
- audit 報告必須附「違反條目修復建議」

## Decision B：預設機制（全局）

### 全局默認值

```
strict = mid
level = P2
最低破壞性原則：默認都偏「不入庫 / 補充案例 / audit」
真要新增需明確信號
```

### 覆蓋標記體系

用戶在採集入口用方括號標記覆蓋默認：

| 標記 | 含義 | 適用採集源 |
|------|------|-----------|
| `[tier=A|B|C]` | 書籍分級 | 書 |
| `[mode=attribution-only|new-rule|audit|extract-only]` | 處理模式 | 復盤/片段/工作產物 |
| `[strict=high|mid|low]` | audit 嚴格度 | 工作產物 |
| `[level=P1|P2|P3]` | 直接指定層級 | 手寫 |
| `[save-as=adr|rule|decision]` | 對話結晶類型 | 對話 |

### 默認動作矩陣

| 來源 | 默認動作 | 默認層級 | 默認配額 |
|------|---------|---------|---------|
| 📚 書 | 全閘審查 | P2 | 3-7/書 |
| 📝 復盤 | 歸因+候選 | P2 | 1-2/篇 |
| ✋ 手寫 | 快速通道 | P2 | 1/次 |
| 🎙️ 片段 | 補充案例優先 | P3 | 5/月全域 |
| 📄 工作產物 | audit 為主 | P2 | 0-2/份 |
| 💬 對話 | L1 摘要+L2 結晶 | P2/P3 | 3/場 + 5 ADR/月 |

詳見 `docs/presets.md`。

## Consequences

**正面：**
- 工作產物從「候選來源」變為「audit 對象」，方向反轉提升規則庫應用價值
- 預設機制大幅降低採集摩擦，用戶不用每次填表
- 覆蓋標記提供精確控制，沒有犧牲靈活性

**負面：**
- 用戶需要記住覆蓋標記體系
- audit 模式可能初期產生大量「規則 vs PRD 衝突」噪音

**緩解：**
- AI 主動提示覆蓋標記
- audit 報告按優先級排序（P1 違反 > P2 違反 > P3 違反）

## Constraints（紅線）

- **不修改 PRD 原文**
- **不寫 rules.json**
- 工作產物默認 strict=mid，需顯式覆蓋才能改

## Implementation

1. 建立 `docs/presets.md` 統一預設機制詳表
2. 採集入口識別覆蓋標記，無標記用默認
3. PRD audit 流程：
   - retrieve rules → 逐節掃描 → 標記 → 報告
4. 報告產出：
   - `docs/audit/[PRD名]-audit-[日期].md` 主報告
   - `docs/review-queue/[PRD名]-candidates-[日期].md` 候選（若有）
   - 簡短摘要：覆蓋率 / P1-P2 違反條數 / 候選提案條數
