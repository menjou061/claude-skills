# ADR-006 手寫快速通道 + 閘 0 元認知（Handwritten Fast-Track + Gate-0 Metacognition）

## Status
Accepted (2026-05-31)

## Context

用戶手寫的規則候選即「精華提煉」——已經過用戶大腦的初步消化，密度高、個人化強。
若強制走完整 7 閘，反而拖慢有價值的提案。

但即使是手寫候選，仍可能有以下質量問題：
- 情緒化（剛踩坑後憤怒寫下，一週後不再認同）
- 重複（與既有規則語義重疊）
- 不可執行（價值觀宣示，缺量化檢查邏輯）
- 缺反例（沒考慮邊界條件，過度泛化）
- 邏輯跳躍（從個別案例直接跳到絕對化結論）

核心矛盾：
- **效率角度**：手寫應該走快速通道，否則打擊用戶提案動力
- **質量角度**：手寫的盲區恰恰是「自以為想清楚了」
- **解法**：減少閘數（快速）+ 加閘 0 元認知（質量自審）

## Decision

### A. 手寫快速通道

手寫候選只過 4 閘：**0 + 1 + 2 + 5**，其餘豁免：

| 閘 | 是否執行 | 說明 |
|----|---------|------|
| 0 元認知 | ✅ | 用戶自審 + AI 質詢 |
| 1 唯一性 | ✅ | AI 對比既有規則 |
| 2 可執行 | ✅ | AI 檢查量化邏輯 |
| 3 痛感 | ⏭️ 豁免 | 手寫即源於痛感，無需證明 |
| 4 頻次 | ⏭️ 豁免 | 用戶判斷頻次，無需 AI 統計 |
| 5 層級 | ✅ | 用戶拍板 P1/P2/P3 |
| 6 回測 | ⏭️ 豁免 | 用戶手寫即代表已內化，無需回測 |

### B. 閘 0 五子項

每條手寫候選必須通過閘 0 元認知：

| 子項 | 內容 | 判斷者 | 失敗動作 |
|------|------|--------|---------|
| A 支撐證據 | ≥ 3 個案例 | 用戶填寫，AI 建議 | < 3 → 標「弱證據」暫存 parking-lot |
| B 情緒中性 | 不是剛踩坑後的情緒輸出 | 用戶自審 | 否 → 建議 24h 冷卻後再提 |
| C 反例/邊界 | 必須寫出反例或邊界條件 | 用戶填寫，**AI 強制質詢** | 缺 → reject |
| D 時間檢驗 | 一週後仍會這麼寫？ | 用戶自審 | 不確定 → 24h 冷卻 |
| E 邏輯閉環 | 無跳躍/無絕對化 | **AI 強制質詢** | 有缺 → 改寫或 reject |

### C. AI 質詢責任分工

- **用戶必填**：A（提供案例）、B（情緒自審）、D（時間自審）
- **AI 強制質詢**：C（反例）、E（邏輯）
- **AI 給建議**：A 證據強度、B/D 提示性檢查

理由：C、E 是用戶最容易自欺的兩項，必須外部質詢；A/B/D 用戶內省即可。

## Consequences

**正面：**
- 手寫提案速度提升（4 閘 vs 7 閘）
- 閘 0 強制反例 + 邏輯閉環，避免「規則庫劣化」最常見失敗模式
- 用戶 mental model 升級：每次手寫都被訓練做反例和邏輯閉環檢查

**負面：**
- 用戶感受「我寫個規則還要被質問反例」可能挫敗
- 閘 0 通過率可能 50% 以下，初期會有大量 reject

**緩解：**
- AI 質詢用「協作語氣」而非「審查語氣」
- 被 reject 的候選進 parking-lot，不丟棄

## Constraints（紅線）

- 手寫候選若繞過閘 0 直接 approve = 違反協議
- 閘 0-C（反例）和閘 0-E（邏輯閉環）AI 必須質詢，不得跳過
- 24h 冷卻期內不得再次提交同一條候選

## Implementation

1. 手寫候選入口：用戶用 `[手寫] [level=P1|P2|P3] [規則名]` 標記
2. AI 立即進入閘 0 質詢流程，輸出 5 子項表格
3. 用戶填完 A/B/D，AI 質詢 C/E
4. 通過閘 0 → 進閘 1/2/5
5. 失敗閘 0 → 進 parking-lot 並標記原因（弱證據/缺反例/邏輯跳躍/情緒未冷卻）
