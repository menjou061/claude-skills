# knowledge-governance SKILL（v1.0）

> 頂層說明文件。三域統一，一套機器，六類採集源，七閘審查。

## 系統職責

以 Obsidian 為敘事真源、Claude Code 為處理器，將個人知識管理從「收藏夾」升級為可驅動決策的閉環系統。詳見 [[docs/PRD.md]]。

## 採集源（6 類）

每類採集源都有默認動作和對應 ADR：

| 來源 | 默認動作 | 配額 | 對應 ADR |
|------|---------|------|---------|
| 📚 書 | 全閘審查（Tier 分級） | 3-7/書 | [[ADR-005]] 書籍分級閱讀協議 |
| 📝 復盤 | 歸因 + 候選 | 1-2/篇 | [[ADR-002]] |
| ✋ 手寫 | 快速通道（4 閘） | 1/次 | [[ADR-006]] 手寫快速通道 + 閘 0 |
| 🎙️ 片段 | 補充案例優先 | 5/月全域 | [[ADR-007]] 精華片段採集機制 |
| 📄 工作產物 | audit 為主（PRD 反向） | 0-2/份 | [[ADR-008]] 工作產物雙向流動 |
| 💬 對話 | L0-L3 四層沉澱 | 3/場 + 5 ADR/月 | [[ADR-009]] 對話即接口 + 結晶 |

詳細配置見 [[docs/presets.md]]。

## 預設機制（最低破壞性原則）

**全局默認：**
- `strict = mid`（audit 嚴格度）
- `level = P2`（候選默認層級）
- **最低破壞性原則**：默認都偏「不入庫 / 補充案例 / audit」，真要新增需明確信號
- **規則庫主權保護 > 提案便利性**

**覆蓋標記體系**（用戶在採集入口用方括號標記覆蓋默認）：

| 標記 | 含義 | 適用採集源 |
|------|------|-----------|
| `[tier=A|B|C]` | 書籍分級 | 書 |
| `[mode=attribution-only|new-rule|audit|extract-only]` | 處理模式 | 復盤/片段/工作產物 |
| `[strict=high|mid|low]` | audit 嚴格度 | 工作產物 |
| `[level=P1|P2|P3]` | 直接指定層級 | 手寫 |
| `[save-as=adr|rule|decision]` | 對話結晶類型 | 對話 |

詳見 [[docs/presets.md]]。

## 對話接口（L0 統一入口）

所有採集源都從對話進入。對話本身也是知識生成過程。

**4 種對話流模式：**

| 模式 | 識別信號 | 處理路徑 |
|------|---------|---------|
| A 貼料流 | 「這篇文章/這本書/這段播客...」 | 對應採集源 SOP |
| B 設計流 | 「我們設計一個 X」「建立 Y 機制」 | 候選 ADR |
| C 諮詢流 | 「我該怎麼做 X」「Y 違反規則嗎」 | retrieve audit |
| D 反思流 | 「我剛踩了個坑」「總結一下我這次...」 | 復盤 SOP |

**4 層沉澱：**

| 層 | 內容 | 持久性 |
|----|------|--------|
| L0 即時 | 對話本身 | 歸零 |
| L1 會話摘要 | 每場 chat 結束自動 | 永久 |
| L2 結構化結晶 | ADR / 規則候選 / SOP | 永久 |
| L3 長期記憶 | Active Threads | 永久 |

詳見 [[docs/conversation-protocol.md]]。

## 規則晉升協議

任何規則晉升必須遵循 [[docs/rule-promotion-process.md]]（7 閘審查 SOP，原 6 閘 + 閘 0 元認知）。

**不允許繞過此流程直接寫 `rules.json`。**

ADR-004 + ADR-006 共同定義晉升流程：
- ADR-004：閘 1-6（唯一性 / 可執行 / 痛感 / 頻次 / 層級 / 回測）
- ADR-006：閘 0 元認知（支撐 / 情緒 / 反例 / 時間 / 邏輯閉環）

核心紅線：
- AI 只能寫 `rules-candidates.json`，絕不碰 `rules.json`
- 規則晉升必須有 Obsidian 歷史案例作為閘 3 證據
- 一輪提案預期晉升 3–7 條，超過必須說明理由
- Tier A 書的規則必須親自讀過對應原文段落
